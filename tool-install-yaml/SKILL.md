# Galaxy Tool-Install YAML

Construct the **tool installation *request* YAML** consumed by ephemeris `shed-tools` and the
`usegalaxy-tools` curation repo. This file names *which Tool Shed repositories to install, at which
revisions, and where to place them in the tool panel*.

**Almost all knowledge for this skill lives in
[`references/tool-install-yaml.md`](references/tool-install-yaml.md)** — a synced white paper
covering the full schema, the consuming tooling, panel-placement semantics, the usegalaxy-tools
manifest/lock pattern, and a field quick-card. **Read that reference before writing anything.** The
notes below are only the fast path.

> Not what you want? Related but distinct artifacts:
> - A single tool's **wrapper `<tool>` XML** (inputs/command/outputs) → see `tool-dev/`.
> - Bumping a revision in an existing **usegalaxy-tools** repo checkout → see `update-usegalaxy-tool/`.
> - This skill is for *authoring/constructing the install YAML itself*.

## Minimal shape

```yaml
tools:
- name: <toolshed repository name>   # REQUIRED — the repo, not the inner tool id
  owner: <toolshed owner>            # REQUIRED — e.g. iuc, devteam, bgruening
  # choose ONE placement key:
  tool_panel_section_label: 'BED tools'   # auto-created if missing; quote multi-word
  # tool_panel_section_id: bed_tools       # must ALREADY exist on the server, else section-less
```

That terse form is all a hand-curated request file needs. Everything else (`revisions`,
`tool_shed_url`, the three `install_*_dependencies` flags) is optional — see the reference for
defaults and the lock-file expansion.

## Decide which mode you're in

| The user wants… | Do this |
|---|---|
| A fresh install file for some named tools | Write the minimal shape above, one entry per repo. Pick `_label` (self-creating) vs `_id` (must pre-exist) per §4 of the reference. |
| Tools needed by a workflow `.ga` | Use ephemeris `workflow-to-tools -w wf.ga -o tools.yml -l <SectionLabel>` (reference §3.5/§6) rather than hand-listing. |
| A snapshot of a live server's tools | Use ephemeris `get-tool-list -g <url> -o out.yml` (reference §3.4). |
| A new section file for the usegalaxy-tools repo | Write only the `.yml` (never the `.yml.lock` by hand); the lock is generated by `make TOOLSET=… fix`. Follow the contributor workflow in reference §5.5. |

## Hard rules (do not get these wrong)

- **`name` is the Tool Shed *repository*, not the inner tool id.** One repo can ship many tools.
- **`tool_panel_section_id` must already exist** in the target server's `shed_tool_conf.xml`;
  otherwise the tool installs section-less (silent failure). Use `tool_panel_section_label` when you
  want the section auto-created.
- **Quote multi-word labels** (`tool_panel_section_label: 'BED tools'`).
- **Never hand-edit `.yml.lock` files** — they are machine-generated from the `.yml`.
- **In usegalaxy-tools, `.yml` files stay terse** (no `revisions`, no dependency flags); pinned
  revisions and `false` dependency flags belong only in the generated lock.
- **Suites:** expand a `suite_*` repo into its component repositories; suites can't be installed via
  the suite repo.

## Validate

If you produced or edited a usegalaxy-tools section file, validate against the repo's pykwalify
schema and regenerate the lock:

```bash
make TOOLSET=<server_name> fix    # or fix-no-deps for pinned/no-dep installs
make TOOLSET=<server_name> lint
```

For a standalone install file, a YAML-parse check plus eyeballing against the §8 quick-card is
enough; the real validation is `shed-tools install` against a server.

## Confirm before acting

Before running any `shed-tools`/`workflow-to-tools`/`get-tool-list` command (these hit live Galaxy
servers / the Tool Shed) or committing files, show the user the YAML you intend to write and the
exact command, and wait for confirmation.
