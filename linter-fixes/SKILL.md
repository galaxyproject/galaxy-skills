# Linting Skill for Galaxy Tools

## Detecting Lint Issues
- Run `uvx planemo lint <tool directory>` to get warnings and errors.
- Pay attention to: ERROR and WARN from the output.
  - Ignore missing citations for now. 

## Fixing Common Problems
1. **XML element order**
   - Reorder top‑level elements so `<macros>` appears before `<xrefs>` (and other top‑level sections) to avoid XMLOrder warnings.
2. **Parameter Naming**
   - Use `argument="--flag"` (or appropriate long flag) so the `name` is generated automatically.
2. **Test Conditionals**
    - Replicate the input `<conditional>` hierarchy in the `<tests>` section using nested `<conditional>` elements.
    - Do **not** include `<when>` tags in tests; the first `<param>` inside each `<conditional>` sets the selected case.
    - Include all required params (including `input_file`) inside the conditional block.
3. **Nested Conditionals**
   - Represent nested conditionals with nested `<conditional>` blocks inside the test, mirroring the tool definition.
4. **Optional Parameters with No Value**
   - Replace empty `value=""` with `value_json="null"` to indicate an omitted optional argument.
5. **Section Grouping**
   - Group related boolean or output‑related parameters inside `<section name="...">` (e.g., `output_options`).
6. **Advanced Params**
    - Wrap advanced‑only parameters inside a `<conditional name="advanced_params">` block.
7. **Boolean Test Parameters**
    - When lint reports test validation errors for boolean params (e.g., `Input should be a valid boolean [type=bool_type]`), do **not** change the `<param>` definition.
    - Instead, change the test’s `<param>` value from `value='--flag'` (or any flag string like `-upgma`, `-kimura`) to `value='true'` (or `value='false'` where appropriate).
    - Example: if a param is defined as `<param name='freq' type='boolean' truevalue='--freq' />`, the test should use `<param name='freq' value='true'/>` — not `value='--freq'`.
    - This satisfies Pydantic validation that expects boolean params to receive `true`/`false`, not flag strings.
8. **Missing number of expected outputs**
    - Add `expect_num_outputs` to the test definition
    - Try to derive the number from the definition of the output filters and the values chosen in the test or from the number of outputs defined
      in the test.

## Validation Workflow
```bash
# 1. Lint the tools in a tool repository 
uvx planemo lint tools/<repository name>/

# 2. Apply fixes (edit XML as described above)
#    – use a text editor or `opencode edit`.

# 3. Re‑run lint to confirm no errors remain.
uvx planemo lint tools/<repository name>/

# 4. Run tests to ensure functionality.
uvx planemo test --biocontainers tools/<repository name>/
```

## Checklist Before Removing `.lint_skip`
- ✅ All `planemo lint` warnings resolved (except acceptable `CitationsNoValid`).
- ✅ All tests pass.
- ✅ No `<when>` tags present in `<tests>`.
- ✅ Optional params use `value_json="null"` when no value is required.
- ✅ Indentation is 4 spaces throughout.
- ✅ `expect_num_outputs` values now match output filters.
- ✅ Test params inside conditionals use `<conditional name="..."><param .../></conditional>` structure matching the tool definition.
- ✅ Conditional param values should match the defined option values (e.g., `value="required"`, `value="full"`, not `value="-flag"`).
- ✅ Boolean test params use `value='true'` or `value='false'` instead of flag strings.
- ✅ `<macros>` appear before `<xrefs>` and other top‑level elements.
